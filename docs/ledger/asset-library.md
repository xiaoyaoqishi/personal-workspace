# Ledger 资产库设计文档

## 1. 模块定位

资产库是 Ledger 域内的独立子模块，用于个人资产全生命周期管理。

资产库集成在账务管理中，但不依赖账务导入、交易流水、复核工作台、商户识别或规则引擎。用户应能直接创建、维护、估值、追踪和复盘资产。

资产库不是普通物品清单，而是个人资产系统。它综合：

- “归物”式资产档案、分类、标签、检索、整理能力；
- “有数”式生命周期、估值、日均成本、闲置、卖出和盈亏复盘能力。

核心目标：让用户知道自己拥有什么、现在值多少、真实花费多少、每天成本多少、最终亏赚多少。

## 2. 功能结构

资产库只有一个一级入口：

- 页面名称：资产库
- 前端归属：frontend-ledger
- 推荐路由：/assets
- 后端命名空间：/api/ledger/assets

资产库主页面通过 Tabs 组织功能，不新增多个侧边栏入口。

V1 页签：

1. 总览
2. 资产列表
3. 新增资产
4. 生命周期
5. 分析
6. 设置

## 3. 核心能力

资产库由四层能力构成：

1. 资产档案层：名称、类型、分类、标签、品牌、型号、序列号、位置、图片、备注。
2. 生命周期层：买入、开始使用、估值、维修、保养、配件、使用打卡、闲置、重新启用、挂售、卖出、退役、报废、丢失。
3. 成本与估值层：买入成本、附加成本、当前估值、净消费成本、日均成本、残值率、卖出收入、卖出盈亏。
4. 分析复盘层：总资产净值、分类分布、状态分布、日均成本排行、闲置排行、残值率排行、已卖出盈亏复盘。

V1 不做保守空壳，要形成“资产新增 -> 资产列表 -> 生命周期事件 -> 估值 -> 成本计算 -> 总览分析”的可用闭环。

## 4. 资产状态

V1 资产状态：

| 状态 | 含义 |
|---|---|
| draft | 草稿 |
| in_use | 使用中 |
| idle | 闲置中 |
| on_sale | 出售中 |
| sold | 已卖出 |
| retired | 已退役 |
| disposed | 已报废 |
| lost | 已丢失 |

原则：

- sold、disposed、lost 是生命周期终态；
- idle 不是终态，仍可重新启用或出售；
- on_sale 是出售前状态；
- 状态变化优先通过生命周期事件驱动。

## 5. 生命周期事件

V1 事件类型：

| 事件 | 含义 |
|---|---|
| purchase | 购入 |
| start_use | 开始使用 |
| valuation | 估值更新 |
| repair | 维修 |
| maintenance | 保养 |
| accessory | 增加配件或附加成本 |
| usage | 使用打卡 |
| idle | 转为闲置 |
| resume | 重新启用 |
| on_sale | 开始挂售 |
| sell | 卖出 |
| retire | 退役 |
| dispose | 报废 |
| lost | 丢失 |
| note | 备注 |

原则：

- 事件是资产历史事实记录；
- 状态、金额、估值变化应可追溯；
- 卖出、报废、丢失事件必须影响资产主状态；
- V1 删除事件可以不自动回滚资产主表，但 UI 需要说明后果。

## 6. 数据对象

V1 三个核心对象：

### Asset

资产主对象，保存稳定信息和当前状态。

建议字段：名称、类型、分类、状态、品牌、型号、序列号、位置、购买渠道、购买日期、开始使用日期、结束日期、买入价格、附加成本、当前估值、卖出价格、目标日均成本、预期使用天数、使用次数、保修到期日、是否计入净资产、标签、图片、备注、owner_role、软删除字段、创建和更新时间。

### AssetEvent

生命周期事件对象，保存资产变化历史。

建议字段：asset_id、event_type、event_date、title、amount、value_after、note、metadata、owner_role、created_at。

### AssetValuation

估值记录对象，保存价值变化历史。

建议字段：asset_id、valuation_date、value、valuation_type、source、note、owner_role、created_at。

V1 可先用字符串分类和 JSON 标签/图片，不必拆 AssetCategory、AssetTag、AssetAttachment。

## 7. 核心指标

单资产需要计算：

- 持有天数
- 使用天数
- 总投入成本
- 净消费成本
- 已实现消费成本
- 现金日均成本
- 净消费日均成本
- 已实现日均成本
- 残值率
- 卖出盈亏
- 目标日均成本进度

口径：

- 总投入成本 = 买入价格 + 附加成本
- 净消费成本 = 总投入成本 - 当前估值
- 已实现消费成本 = 总投入成本 - 卖出价格
- 现金日均成本 = 总投入成本 / 使用天数
- 净消费日均成本 = 净消费成本 / 使用天数
- 已实现日均成本 = 已实现消费成本 / 使用天数
- 残值率 = 当前估值 / 总投入成本
- 卖出盈亏 = 卖出价格 - 总投入成本

计算必须容错：日期为空、金额为空、使用天数为 0 都不能导致报错。无法计算时返回 null 或约定默认值。

## 8. 后端边界

资产库后端归属 Ledger 域。

推荐结构：

- backend/models/ledger.py
- backend/schemas/ledger_assets.py
- backend/services/ledger/assets_service.py
- backend/routers/ledger_assets.py

原则：

- API 命名空间使用 /api/ledger/assets；
- Router 只做参数接收、依赖注入和响应封装；
- 领域逻辑放在 ledger service 或 dedicated runtime；
- 不把资产库逻辑写入 backend/main.py 或 backend/services/runtime.py；
- 不依赖账务导入、复核、商户识别、规则引擎；
- 所有数据按 owner_role 隔离；
- 删除采用软删除。

## 9. 前端边界

资产库前端归属 frontend-ledger。

推荐结构：

- frontend-ledger/src/pages/AssetsPage.jsx
- frontend-ledger/src/api/assets.js
- frontend-ledger/src/components/assets/
- frontend-ledger/src/styles/asset-library.css

原则：

- 只新增一个「资产库」侧边栏入口；
- 所有功能通过 Tabs 承载；
- 使用现有 React、Ant Design、Recharts；
- 不引入新的 UI 框架；
- 样式 className 使用 asset-library 或 ledger-assets 前缀，避免污染其他页面。

## 10. UI/UX 原则

资产库 UI 要强调信息密度、可读性、状态清晰、关键指标突出、操作后果明确。

要求：

- 总览页像个人资产仪表盘；
- 资产列表同时支持卡片视图和表格视图；
- 新增资产表单分区，不做超长表单；
- 生命周期使用时间线展示；
- 分析页服务决策，不只是堆图表；
- 空状态、加载状态、错误状态明确；
- 删除、卖出、报废、丢失必须有确认文案。

## 11. V1 范围

V1 必须做：

- 资产新增、编辑、软删除、列表检索、详情查看；
- 估值记录；
- 生命周期事件；
- 事件驱动状态更新；
- 日均成本、净消费成本、残值率、卖出盈亏计算；
- 总览基础统计；
- 分析页关键复盘指标；
- 资产库入口和 Tabs 页面；
- 完整 UI，不做后台 CRUD 半成品。

V1 暂不做：

- OCR
- 电商订单导入
- 市场价格自动抓取
- 多人协作
- 云同步
- 复杂折旧模型
- 和交易流水自动生成资产

## 12. 分阶段计划

1. Phase 1：后端资产模型、API、service、summary、指标计算。
2. Phase 2：前端资产库入口、/assets 页面、Tabs 主结构。
3. Phase 3：资产新增、资产列表、详情、编辑、软删除。
4. Phase 4：生命周期事件、估值记录、状态流转。
5. Phase 5：总览、分析、排行榜、盈亏复盘。
6. Phase 6：UI 美化、空状态、错误状态、构建与测试验收。

## 13. 验收标准

每阶段完成后输出：

- 修改文件列表；
- 本阶段实现内容；
- 未做事项；
- 验证方式；
- 是否影响现有 Ledger 功能。

最终需要确认：

- 现有导入中心、复核工作台、商户词典、规则管理、基础分析仍可用；
- 资产库入口可见，Tabs 可切换；
- 可以新增、查看、编辑、软删除资产；
- 可以新增生命周期事件和估值；
- 卖出、报废、丢失事件能正确更新状态和指标；
- 总览和分析能展示关键指标；
- 前端构建通过；
- 后端测试通过或说明已有失败原因。

## 14. 后续增强方向

后续可考虑：

- 和账务流水手动关联；
- 大额支出生成资产候选；
- 图片文件上传；
